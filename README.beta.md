# awg-tgbot (beta)

Beta-ветка Telegram-бота для продажи доступа к **уже установленному** AmneziaWG (AWG) в self-hosted формате.

> `beta` нужна для ранней проверки улучшений в надёжности выдачи, обработке платежей и защите от рассинхронизаций.  
> Если вам нужна максимально консервативная ветка — используйте `main`.

## Для кого эта ветка

- для администраторов, готовых тестировать новые механизмы раньше `main`;
- для тех, кому важна повышенная устойчивость payment/provisioning flow;
- для пилотирования на тестовом или аккуратно контролируемом проде.

## Что важно знать сразу

- `beta` не равна «нестабильная любой ценой», но изменения приходят сюда раньше;
- базовые требования те же: Ubuntu/Debian, root, Docker, systemd, рабочий AWG контейнер;
- архитектурная модель остаётся single-node + SQLite.

## Быстрый старт (beta)

### 1) Что подготовить

- сервер Ubuntu/Debian;
- root-доступ;
- Docker;
- заранее развернутый и рабочий контейнер AmneziaWG/AWG;
- Telegram bot token;
- `ADMIN_ID`.

### 2) Установка из `beta`

```bash
curl -fsSL https://raw.githubusercontent.com/Just1k13/awg-tgbot/beta/awg-tgbot.sh | sudo REPO_BRANCH=beta bash
```

или

```bash
wget -qO- https://raw.githubusercontent.com/Just1k13/awg-tgbot/beta/awg-tgbot.sh | sudo REPO_BRANCH=beta bash
```

### 3) Проверка запуска

```bash
sudo systemctl status vpn-bot.service --no-pager
sudo journalctl -u vpn-bot.service -n 120 --no-pager
```

## Чем `beta` отличается от `main`

`beta` сфокусирована на улучшениях надёжности и безопасной эксплуатации:

1. **Preflight-проверка ёмкости** перед покупкой/подтверждением платежа.
2. **Более явная модель статусов платежей** и идемпотентная повторная обработка.
3. **Recovery после рестарта** для незавершённых payment/provisioning операций.
4. **Более безопасный provisioning flow**, чтобы снизить риск рассинхронизации AWG ↔ SQLite.
5. **Строже поведение revoke/delete** при частичных сбоях удаления peer.

Это делает ветку удобной для раннего внедрения защитных механизмов без смены бизнес-флоу (Telegram Stars и текущая тарифная логика сохраняются).

## Что умеет проект

### Пользователь

- покупка доступа через Telegram Stars;
- продление подписки;
- получение до `CONFIGS_PER_USER` конфигов;
- выдача `vpn://` и `.conf`;
- понятный ответ при нехватке слотов;
- профиль + инструкция + поддержка.

### Администратор

- админ-панель в Telegram;
- просмотр статистики, audit, orphan peers;
- ручная выдача/отзыв/удаление;
- sync AWG ↔ база;
- backup БД без секретов;
- инструменты для безопасной повторной обработки после сбоев.

### Сервер

- единый installer/updater/remover-скрипт;
- systemd-сервис;
- SQLite + миграции;
- шифрование чувствительных данных;
- логирование и базовая диагностика на старте.

## Надёжный flow покупки и выдачи

Высокоуровнево в `beta`:

1. Пользователь выбирает тариф.
2. Перед оплатой проверяется доступная ёмкость (слоты/IP).
3. Платёж фиксируется со статусом обработки.
4. Выдача идёт через защищённые provisioning-состояния.
5. При дублях webhook/updates повторная обработка не должна ломать логику.
6. После рестарта незавершённые операции поднимаются и повторно сверяются.

> Важно: это не магическая гарантия от любых аварий, а практический набор механизмов для self-hosted single-node сценария.

## Установка и переключение веток

### Обновление текущей ветки

```bash
sudo awg-tgbot update
```

### Проверка обновлений

```bash
sudo awg-tgbot check-updates
```

### Переключение между `main` и `beta`

```bash
sudo REPO_BRANCH=main awg-tgbot update
sudo REPO_BRANCH=beta awg-tgbot update
```

Выбранная ветка хранится в `/opt/amnezia/bot/.state/repo_branch`.

## Конфигурация (.env)

### Обязательные

- `API_TOKEN`
- `ADMIN_ID`
- `ENCRYPTION_SECRET`

### Интеграция с AWG

- `DOCKER_CONTAINER`
- `WG_INTERFACE`
- `SERVER_PUBLIC_KEY`
- `SERVER_IP`

### Бизнес-параметры

- `STARS_PRICE_7_DAYS`
- `STARS_PRICE_30_DAYS`
- `CONFIGS_PER_USER`

### Ёмкость и сеть

- `VPN_SUBNET_PREFIX`
- `FIRST_CLIENT_OCTET`
- `MAX_CLIENT_OCTET`

> В `beta` особое внимание уделяйте диапазону пула и фактической ёмкости: именно эти параметры влияют на preflight-check и поведение при sold out.

## Безопасность и данные

- чувствительные данные хранятся в зашифрованном виде;
- `ENCRYPTION_SECRET` обязателен для корректной работы с уже сохранёнными секретами;
- backup-функция ориентирована на выгрузку БД без секретов;
- при ручных AWG-правках есть риск рассинхронизации с SQLite;
- в `beta` добавлены дополнительные шаги reconciliation/recovery, но ручные операции всё равно требуют аккуратности.

---

## Ограничения beta

- это не оркестратор multi-node/HA;
- SQLite остаётся single-node storage;
- поведение зависит от доступности Docker и AWG;
- beta-функции могут менять внутренние детали обработки раньше, чем попадут в `main`.

## Troubleshooting / FAQ

### `Unauthorized` от Telegram

Проверьте `API_TOKEN` и перезапустите сервис.

### Бот не стартует после обновления

```bash
sudo systemctl status vpn-bot.service --no-pager
sudo journalctl -u vpn-bot.service -n 200 --no-pager
```

Проверьте `.env`, контейнер, интерфейс AWG.

### Платёж принят, а выдача задерживается

Проверьте логи сервиса: в `beta` поддерживается recovery незавершённых операций после рестарта/сбоев.

### После миграции/переноса БД что-то сломалось

Сначала сверьте `ENCRYPTION_SECRET` и сетевые параметры пула, затем проверьте синхронизацию AWG ↔ SQLite.
