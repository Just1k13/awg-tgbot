# awg-tgbot

Удобный Telegram-бот для продажи доступа и автоматической выдачи конфигов **AmneziaWG / AWG** на **уже работающем** self-hosted сервере.

Проект помогает не поднимать VPN с нуля, а аккуратно **подключить бота к существующему AWG**, настроить установку, обновления и повседневное администрирование.  
После этого пользователь покупает доступ через **Telegram Stars**, а бот сам продлевает подписку и выдаёт конфиги.

> [!IMPORTANT]
> `awg-tgbot` **не разворачивает AmneziaWG с нуля**.  
> Сначала у вас должен быть **уже установлен и работающий контейнер AmneziaWG / AWG**, и только потом имеет смысл ставить этот проект.

---

## Основные возможности

### Что получает пользователь
- покупку доступа через **Telegram Stars**;
- продление уже активной подписки;
- профиль со сроком действия доступа;
- выдачу до `CONFIGS_PER_USER` конфигов на пользователя;
- `vpn://` ключ для быстрого импорта в Amnezia;
- `.conf` файл для ручного подключения;
- инструкцию по подключению;
- кнопку связи с поддержкой.

### Что получает администратор
- встроенную админ-панель в боте;
- статистику по пользователям, ключам и свободным IP;
- список последних пользователей;
- ручную выдачу доступа;
- отзыв доступа;
- полное удаление пользователя;
- проверку синхронизации **AWG ↔ БД**;
- безопасную работу с orphan peer только в домене ключей, созданных ботом;
- журнал действий (`audit_log`);
- массовую рассылку;
- резервную копию БД **без секретов**.

### Что получает сервер
- единый скрипт для установки, обновления, переустановки, удаления и диагностики;
- безопасный `preflight` перед любыми действиями;
- автоопределение контейнера AWG, интерфейса, публичного ключа сервера и endpoint;
- `systemd`-сервис для автозапуска;
- запуск бота от отдельного системного пользователя;
- работу с AWG через ограниченный helper, без добавления бота в `docker` group;
- хранение текущей ветки (`main` или `beta`) и безопасный update только по pinned commit SHA.

## Как это работает

Если по-простому, схема такая:

1. Пользователь открывает бота и выбирает покупку доступа.
2. Оплачивает тариф на **7** или **30** дней через **Telegram Stars**.
3. Бот:
   - принимает платёж;
   - активирует или продлевает подписку;
   - при необходимости создаёт недостающие peer в AWG;
   - сохраняет клиентские секреты в зашифрованном виде;
   - отдаёт пользователю `vpn://` и `.conf`.
4. В фоне бот следит за просроченными подписками, зависшими платежами и служебными состояниями.
5. Администратор при необходимости вмешивается через команды и админ-панель.

> Идея проекта простая: **обычный пользователь получает доступ без ручной переписки с админом**, а администратор — меньше рутины и меньше шансов что-то сломать руками.

## Что нужно перед установкой

| Что нужно | Зачем это нужно |
|---|---|
| **Ubuntu / Debian** | это целевая среда installer-скрипта |
| **root-доступ** | для установки, обновления, `systemd`, helper и служебных файлов |
| **Docker** | AWG работает через контейнер |
| **Уже работающий контейнер AmneziaWG / AWG** | проект подключается к нему, а не разворачивает его |
| **Токен Telegram-бота** | без него бот не запустится |
| **Telegram `user_id` администратора** | чтобы открыть админ-доступ |
| **Обычная интерактивная SSH / console-сессия** | prompt-режимы installer требуют TTY |

### Что installer обычно определяет сам
Скрипт старается автоматически найти:
- контейнер AWG;
- интерфейс (`awg0` и т.п.);
- путь к конфигу внутри контейнера;
- `SERVER_PUBLIC_KEY`;
- внешний `SERVER_IP`;
- `PUBLIC_HOST`;
- часть параметров AWG.

### Что обычно вводится руками
Обычно вручную нужны только:
- `API_TOKEN`;
- `ADMIN_ID`;
- `SERVER_NAME`.

## Быстрый старт

> [!TIP]
> В командах ниже используется `set -euo pipefail`, чтобы скачивание не заканчивалось «тихо» при ошибке сети или пустом вводе.

### Установка `main`

```bash
set -euo pipefail
curl -fsSL https://raw.githubusercontent.com/Just1k13/awg-tgbot/main/awg-tgbot.sh | sudo bash -s --
```

После запуска скрипт:
1. выполнит безопасную предварительную проверку;
2. покажет краткую сводку;
3. откроет меню;
4. начнёт установку **только после вашего явного выбора**.

> Вместо `curl` можно использовать `wget -qO-` с тем же URL.

---

### Установка `beta`

Для первой установки `beta` используйте ту же команду, но с `REPO_BRANCH=beta`:

```bash
set -euo pipefail
curl -fsSL https://raw.githubusercontent.com/Just1k13/awg-tgbot/beta/awg-tgbot.sh | sudo REPO_BRANCH=beta bash -s --
```

После первой установки выбранная ветка сохраняется автоматически в:

```text
/opt/amnezia/bot/.state/repo_branch
```

Дальше обновление выполняется только с явным pinned SHA:

```bash
sudo REPO_UPDATE_REF=<40-hex-commit-sha> awg-tgbot update
```

---

### Переключение между ветками

Перейти с `beta` на `main`:

```bash
sudo REPO_BRANCH=main awg-tgbot
```

Вернуться с `main` на `beta`:

```bash
sudo REPO_BRANCH=beta awg-tgbot
```

После переключения ветка сохраняется в state и далее используется для `check-updates`.

---

### Что сделать после установки

Проверьте базовые вещи:

```bash
sudo awg-tgbot preflight
sudo awg-tgbot status
systemctl status vpn-bot.service --no-pager -l
```

Что вы хотите увидеть:
- AWG определяется как найденный;
- бот установлен;
- сервис `vpn-bot.service` запущен;
- в статусе отображается правильная ветка;
- в `.env` есть основные значения вроде `SERVER_PUBLIC_KEY` и `SERVER_IP`.

## Основные команды

### Самое полезное на каждый день

| Команда | Что делает |
|---|---|
| `sudo awg-tgbot` | обычный запуск: `preflight` → краткая сводка → меню |
| `sudo awg-tgbot preflight` | безопасная предварительная проверка окружения |
| `sudo awg-tgbot detect-install-state` | то же самое, альтернативное имя команды |
| `sudo awg-tgbot status` | показывает статус установки, сервиса, ветки, логов и AWG target |
| `sudo awg-tgbot check-updates` | проверяет, есть ли обновление для текущей ветки |
| `sudo REPO_UPDATE_REF=<40-hex-sha> awg-tgbot update` | обновляет на явно заданный immutable commit |
| `sudo awg-tgbot reinstall` | запускает переустановку |
| `sudo awg-tgbot remove` | открывает меню удаления |
| `sudo awg-tgbot sync-helper-policy` | синхронизирует helper policy после ручных правок `.env` |

### Если нужен более ручной режим

| Команда | Когда полезна |
|---|---|
| `sudo awg-tgbot install` | если хотите явно запустить установку как action |
| `sudo awg-tgbot logs` | если хотите открыть меню логов и диагностики |
| `sudo awg-tgbot remove-default` | удалить код и сервис, но оставить `.env` и БД |
| `sudo awg-tgbot remove-full` | полное удаление всего проекта |

> [!IMPORTANT]
> Команды, где нужен ввод, подтверждение или меню, требуют **TTY**.  
> Для обычной работы запускайте их в нормальной SSH-сессии или через локальную консоль.

### Полезные команды проверки

```bash
systemctl status vpn-bot.service --no-pager -l
journalctl -u vpn-bot.service -f
tail -f /var/log/awg-tgbot/bot.log
```

- `systemctl status` — состояние сервиса;
- `journalctl` — systemd-логи;
- `tail -f` — лог самого приложения.

## Что умеет бот

### Для пользователя

Пользователь видит понятный интерфейс без лишней технички:

- запуск через `/start`;
- покупка доступа на **7** или **30** дней;
- продление активной подписки;
- просмотр профиля и срока действия;
- получение `vpn://` для быстрого импорта;
- получение `.conf` для ручного подключения;
- кнопка с инструкцией;
- кнопка поддержки.

Иными словами, для обычного человека это выглядит как:  
**открыл бота → оплатил → получил доступ → подключился**.

---

### Для администратора

Администратор получает управление прямо в Telegram:

- админ-панель внутри бота;
- общую статистику;
- список пользователей с пагинацией;
- карточку пользователя с быстрыми действиями;
- выдачу доступа на 1 / 7 / 30 дней;
- отзыв доступа;
- удаление пользователя;
- проверку синхронизации AWG и БД;
- работу с orphan peer;
- аудит действий;
- рассылку;
- безопасный backup базы без секретов.

### Основные админ-команды

```text
/give ID [ДНИ]
/revoke ID
/users
/stats
/orphans
/audit [LIMIT]
/sync_awg
/clean_orphans
/clean_orphans_force
/backup
/send ТЕКСТ
```

Коротко:
- `/give` — выдать или продлить доступ;
- `/revoke` — отключить пользователя;
- `/users` — последние пользователи;
- `/stats` — общая статистика;
- `/orphans` — посмотреть orphan peer;
- `/audit` — журнал действий;
- `/sync_awg` — быстрая проверка AWG ↔ БД;
- `/clean_orphans` — quarantine-этап только для `bot_managed` peer, без физического удаления;
- `/clean_orphans_force` — физическое удаление только `bot_managed` peer после двойного подтверждения;
- `/backup` — резервная копия БД без секретных данных;
- `/send` — массовая рассылка.

## Важные замечания

### 1) Проект всегда начинает с безопасной проверки
Обычный запуск:

```bash
sudo awg-tgbot
```

не должен самовольно:
- ставить пакеты;
- менять `.env`;
- трогать `systemd`;
- начинать установку без вашего выбора.

Сначала идёт только **read-only проверка**, потом — меню.

---

### 2) `SERVER_NAME` — это имя, а не endpoint
`SERVER_NAME` нужен только как отображаемое имя VPN.  
Он **не участвует** в сборке endpoint.

Для подключения используется `SERVER_IP` в формате:

```text
IPv4:port
```

### 3) Fail-fast валидация критичных сетевых настроек
Бот больше не стартует с «мусорными» значениями:
- `PERSISTENT_KEEPALIVE`: только `0`/`off` или диапазон `1..65535`;
- `CLIENT_ALLOWED_IPS`: только список CIDR через запятую (`0.0.0.0/0, ::/0` и т.п.);
- `SERVER_IP` / `PUBLIC_HOST`: только валидный публичный IPv4 (для `SERVER_IP` — с портом).

Если значение некорректно — запуск завершается сразу с понятной ошибкой.

### 4) Безопасный update flow
Обновление по mutable ветке без фиксации версии больше не выполняется.

Команда `sudo awg-tgbot update` теперь требует pinned commit SHA:

```bash
sudo REPO_UPDATE_REF=<40-hex-commit-sha> awg-tgbot update
```

Если `REPO_UPDATE_REF` не задан — update будет остановлен. Это сделано специально, чтобы не было «слепого» деплоя tarball ветки под root.

---

### 3) Для endpoint нужен внешний IPv4
Проект рассчитывает на внешний endpoint именно в формате **IPv4:port**.

Это значит:
- домен как endpoint здесь не основной сценарий;
- hostname вместо внешнего IPv4 не используется;
- если автоопределение не сработало, адрес придётся ввести вручную.

---

### 4) `API_TOKEN` и `ADMIN_ID` автоматически не подставляются
Эти значения installer не угадывает.  
Их нужно ввести вручную при установке или переустановке.

---

### 5) `ENCRYPTION_SECRET` нужно беречь
Installer умеет сгенерировать `ENCRYPTION_SECRET`, если его ещё нет.  
Но после этого его важно **сохранить и не потерять**.

Зачем это нужно:
- клиентские приватные ключи и PSK в БД хранятся в зашифрованном виде;
- без старого `ENCRYPTION_SECRET` вы не сможете расшифровать уже сохранённые данные;
- при переносе БД на другой сервер этот секрет тоже нужен.

---

### 6) Если вручную меняли `.env`, синхронизируйте policy
Если вы руками изменили:
- `DOCKER_CONTAINER`
- `WG_INTERFACE`

после этого выполните:

```bash
sudo awg-tgbot sync-helper-policy
sudo awg-tgbot status
```

Иначе runtime и helper policy могут смотреть в разные AWG target.

---

### 8) Какие переменные в `.env` важны для runtime

Ключевые runtime-переменные (`API_TOKEN`, `ADMIN_ID`, `SERVER_PUBLIC_KEY`, `SERVER_IP`, `DOCKER_CONTAINER`, `WG_INTERFACE`) должны оставаться валидными.

Дополнительно поддерживаются:
- `AWG_HELPER_PATH`
- `AWG_HELPER_USE_SUDO`
- `AWG_HELPER_POLICY_PATH`
- `PENDING_KEY_TTL_SECONDS`
- `PAYMENT_RETRY_DELAY_SECONDS`
- `PAYMENT_PROVISIONING_LEASE_SECONDS`

Installer сам поддерживает в актуальном состоянии helper/policy/sudoers файлы; после ручных правок `DOCKER_CONTAINER`/`WG_INTERFACE` всегда выполняйте `sync-helper-policy`.

## FAQ

### Нужно ли сначала установить AWG?
Да. Это обязательное условие.

Если на сервере нет работающего контейнера AmneziaWG / AWG, installer это увидит и не будет делать вид, что всё в порядке.

---

### Что installer спросит у меня при установке?
В типичном случае:
- токен бота;
- `ADMIN_ID`;
- имя сервера.

Остальное он постарается определить сам.  
Если окружение нестандартное, можно выбрать ручной режим и явно задать:
- контейнер;
- интерфейс;
- `SERVER_PUBLIC_KEY`;
- `PUBLIC_HOST`;
- `SERVER_IP`;
- цены в Stars;
- `DOWNLOAD_URL`;
- `SUPPORT_USERNAME`.

---

### Как понять, что всё действительно запустилось?
Сделайте так:

```bash
sudo awg-tgbot preflight
sudo awg-tgbot status
systemctl status vpn-bot.service --no-pager -l
journalctl -u vpn-bot.service -n 50 --no-pager
```

Хороший результат выглядит так:
- AWG найден;
- бот установлен;
- `vpn-bot.service` в `active (running)`;
- в логах нет `Unauthorized` и ошибок запуска;
- текущая ветка отображается корректно.

---

### Можно ли потом перейти с `main` на `beta` и обратно?
Да.

На `main`:

```bash
sudo REPO_BRANCH=main awg-tgbot
```

На `beta`:

```bash
sudo REPO_BRANCH=beta awg-tgbot
```

---

### Что делать, если бот пишет `Unauthorized`?
Обычно это означает, что токен Telegram-бота неверный или устарел.

Что делать:
1. перевыпустить токен в **BotFather**;
2. обновить `API_TOKEN` в `.env`;
3. перезапустить сервис или выполнить обновление/переустановку.

---

### Что происходит при удалении?
Есть два режима.

**Обычное удаление**
- удаляет код, сервис, venv и логи;
- оставляет `.env` и БД.

**Полное удаление**
- удаляет всё, включая БД и `.env`;
- перед этим пытается удалить из AWG peer, созданные ботом и найденные в текущей БД;
- если удаление хотя бы одного peer завершается ошибкой, полное удаление останавливается.

Это полезно, чтобы не потерять БД раньше времени и не оставить систему в странном состоянии.

---

### Можно ли использовать проект без installer?
Технически да: код бота можно запускать отдельно.

Но нормальный сценарий проекта — использовать штатный installer, потому что он:
- готовит окружение;
- ставит зависимости;
- создаёт сервис;
- настраивает helper;
- синхронизирует служебные файлы.
