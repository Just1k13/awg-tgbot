# Аудит awg-tgbot (2026-03-30)

## Критические баги

1. **Потенциальная рассинхронизация состояния peers после рестарта AWG**
   - Бот добавляет/удаляет peers через `awg set` (runtime), но в коде бота нет явного сохранения в файл конфигурации (`saveconf`) или полной реконцииляции активных peers при старте из БД.
   - Если окружение не сохраняет runtime-состояние интерфейса автоматически, после рестарта контейнера можно потерять активные peers в интерфейсе при сохранении записей в БД.
   - Риск: массовая деградация доступности VPN после рестарта.

2. **Слабая устойчивость парсинга `awg show` → риск неверного учёта занятых IP**
   - Парсер учитывает IP только по шаблону `${VPN_SUBNET_PREFIX}x/32`.
   - При нестандартном формате вывода/AllowedIPs IP может не распознаться (`ip=None`), из-за чего `get_used_ips_from_awg()` не увидит занятый адрес.
   - Риск: повторное выделение IP и сетевые конфликты peer'ов.

## Логические недоработки

1. **Неочевидная модель жизненного цикла `delete_pending`**
   - Для `delete_pending` IP исключаются из `reserved`-набора на этапе выдачи новых ключей.
   - В норме это компенсируется `awg show`, но при частичной деградации AWG/парсинга возможно преждевременное переиспользование IP.

2. **Неатомарность «применить peer + применить QoS»**
   - При удалении доступа peer удаляется, затем очищается QoS.
   - Если QoS-очистка падает, операция классифицируется как ошибка удаления peer и может оставлять «мусорные» tc-правила.

3. **Использование глобальных in-memory limiter-структур**
   - `admin_command_rate_limit` и `purchase_rate_limit` частично очищаются, но при очень большой кардинальности пользователей всё равно возможен рост памяти.
   - Для долгоживущего процесса лучше TTL-cache с ограничением размера (аналогично реализованному в middleware duplicate guard).

## Советы по оптимизации

1. **Ввести startup reconciliation `DB -> AWG` для active keys**
   - На старте сравнивать `keys(state=active)` с `awg show`.
   - Автоматически восстанавливать отсутствующие peers (и QoS), а не только pending-состояние.

2. **Ужесточить и расширить парсер `awg show`**
   - Парсить все `allowed ips` через CIDR-парсер и фильтровать по подсети, а не regex-сопоставлением по префиксу строки.
   - Логировать аномальные peer-записи отдельно.

3. **Асинхронные/неблокирующие операции в handlers**
   - Вынести тяжёлые операции backup, файловые операции и генерацию QR в `asyncio.to_thread()` или воркер.
   - Это снизит задержки обработки событий Telegram в пиковые моменты.

4. **Сделать «операционный kill-switch» для платежей**
   - Добавить флаг «maintenance mode for payments», чтобы быстро отключать pre-checkout при инцидентах инфраструктуры без выката кода.

5. **Усилить политику секретов**
   - Проверять формат `BACKUP_ENCRYPTION_KEY` на старте (валидный Fernet key).
   - Ввести ротацию/версионирование `ENCRYPTION_SECRET` (частично уже реализовано через old secrets).

## Что уже сделано хорошо

- Команды к ОС выполняются без shell-интерполяции (list args), что сильно снижает риск command injection.
- Вспомогательный `awg_helper.py` валидирует container/interface/public key/IP и работает через policy-файл с проверкой прав.
- Приватные ключи/PSK в БД шифруются (`enc:v2` + PBKDF2 + salt).
- Админские команды и callback-потоки защищены проверкой `ADMIN_ID`.
- Есть retry/lease-модель для платежной обработки и журналирование аудита.

## Вердикт

**Текущее состояние: «условно готов», но не production-ready без доработок по отказоустойчивости AWG-state и улучшенного reconciliation.**

Для production рекомендовано:
1) закрыть риски рассинхронизации `DB <-> AWG` после рестартов,
2) укрепить парсинг/учёт IP,
3) вынести блокирующие операции из event loop,
4) добавить операционные защиты (maintenance/health gates) для платежей и provisioning.
